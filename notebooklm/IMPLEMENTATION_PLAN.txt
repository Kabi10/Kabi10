# SL Agrimarket — Full Integration & Play Store Ship Plan

## Context

Audit revealed that while builds pass and core commerce flows (listings, transactions) have backend integration, **5 features are UI shells with no backend**:
- Chat/Messaging (Room-only, no backend endpoints exist)
- Favorites (Room-only)
- Reviews (Room-only)
- Notifications (Room-only, no FCM)
- Profile editing (local-only, backend endpoint exists but app never calls it)

Additionally, there are **critical backend security issues** (SQL injection, insecure OTP generation, mock SMS) that must be fixed before shipping.

This plan covers: backend fixes → new backend endpoints → Android wiring → real-time chat → Play Store prep.

---

## PHASE 1: Backend Security Fixes (Critical)

### 1A. Fix SQL injection in listings route
**File:** `backend/src/routes/listings.js` (~line 146)
- `sortBy` parameter is interpolated directly into SQL: `ORDER BY l.${sortBy}`
- Fix: whitelist allowed sort columns (`created_at`, `price_per_unit`, `quantity`, `updated_at`, `view_count`)
- Default to `created_at` if invalid value

### 1B. Fix insecure OTP generation
**File:** `backend/src/utils/helpers.js` (~line 8)
- Replace `Math.random()` with `crypto.randomInt(100000, 999999)`
- Import `crypto` from Node.js built-in

### 1C. Uncomment DB health check
**File:** `backend/src/server.js` (~lines 191-193)
- Uncomment `await database.query('SELECT NOW()')`
- Add try/catch with graceful fallback (log warning but don't crash — for Vercel cold starts)

### 1D. Configure Twilio SMS
**File:** `backend/.env` (production env vars on Vercel)
- Set `SMS_MODE=twilio`
- Set `TWILIO_ACCOUNT_SID`, `TWILIO_AUTH_TOKEN`, `TWILIO_PHONE_NUMBER`
- The Twilio implementation already exists in `backend/src/services/smsService.js`
- User provides Twilio credentials; we configure the env vars on Vercel

### 1E. Fix Supabase silent fallback
**File:** `backend/src/config/supabase.js` (~lines 14-15)
- In production (`NODE_ENV=production`), throw error instead of falling back to `https://example.supabase.co`

## PHASE 2: Database Schema Updates

**File:** `backend/src/database/schema.sql` — add new tables:

### 2A. `conversations` table
```sql
- id UUID PK
- participant_1_id UUID FK -> users(id)
- participant_2_id UUID FK -> users(id)
- listing_id UUID FK -> listings(id) (nullable, context for conversation)
- last_message_text TEXT
- last_message_at TIMESTAMPTZ
- created_at, updated_at TIMESTAMPTZ
- UNIQUE(participant_1_id, participant_2_id) with ordered check
```

### 2B. `messages` table
```sql
- id UUID PK
- conversation_id UUID FK -> conversations(id)
- sender_id UUID FK -> users(id)
- content TEXT NOT NULL
- message_type VARCHAR(20) DEFAULT 'TEXT'
- is_read BOOLEAN DEFAULT FALSE
- created_at TIMESTAMPTZ
```

### 2C. `favorites` table
```sql
- id UUID PK
- user_id UUID FK -> users(id)
- listing_id UUID FK -> listings(id)
- created_at TIMESTAMPTZ
- UNIQUE(user_id, listing_id)
```

### 2D. `reviews` table
```sql
- id UUID PK
- transaction_id UUID FK -> transactions(id)
- reviewer_id UUID FK -> users(id)
- reviewee_id UUID FK -> users(id)
- rating INTEGER CHECK 1-5
- comment TEXT
- review_type VARCHAR(20) CHECK IN ('BUYER_TO_FARMER', 'FARMER_TO_BUYER')
- created_at, updated_at TIMESTAMPTZ
- UNIQUE(transaction_id, reviewer_id)
```

### 2E. `notifications` table
```sql
- id UUID PK
- user_id UUID FK -> users(id)
- type VARCHAR(50) NOT NULL
- title VARCHAR(255) NOT NULL
- message TEXT
- related_id UUID (nullable, polymorphic ref to listing/transaction/message)
- is_read BOOLEAN DEFAULT FALSE
- created_at TIMESTAMPTZ
```

### 2F. `market_prices` table (currently missing from schema but queried by route)
```sql
- id UUID PK
- crop_type VARCHAR(100)
- crop_name_tamil, crop_name_english, crop_name_sinhala VARCHAR(255)
- current_price, previous_price DECIMAL(10,2)
- unit VARCHAR(20)
- trend VARCHAR(20)
- change_percentage, change_amount DECIMAL(10,2)
- location, location_tamil, location_sinhala VARCHAR(255)
- last_updated TIMESTAMPTZ
- is_active BOOLEAN DEFAULT TRUE
- source VARCHAR(50) DEFAULT 'MARKET'
- reliability DECIMAL(3,2) DEFAULT 0.80
```

## PHASE 3: New Backend Routes

Follow existing patterns: `db.query()` via `connection.js`, express-validator, camelCase responses, standard pagination.

### 3A. Messages routes — `backend/src/routes/messages.js`
Mount: `apiRouter.use('/messages', authenticateToken, messagesRoutes)`

| Method | Path | Description |
|--------|------|-------------|
| GET | `/conversations` | List user's conversations (paginated, ordered by last_message_at) |
| GET | `/conversations/:id` | Get conversation detail with participant info |
| POST | `/conversations` | Create or get existing conversation (by participant pair + optional listing) |
| GET | `/conversations/:id/messages` | Get messages in conversation (paginated, ordered by created_at desc) |
| POST | `/conversations/:id/messages` | Send a message (creates notification for recipient) |
| PATCH | `/conversations/:id/read` | Mark all messages in conversation as read |
| GET | `/unread-count` | Get total unread message count for user |

### 3B. Favorites routes — `backend/src/routes/favorites.js`
Mount: `apiRouter.use('/favorites', authenticateToken, favoritesRoutes)`

| Method | Path | Description |
|--------|------|-------------|
| GET | `/` | List user's favorites (paginated, JOINed with listings) |
| POST | `/` | Add listing to favorites (body: { listingId }) |
| DELETE | `/:listingId` | Remove listing from favorites |
| GET | `/check/:listingId` | Check if listing is favorited |

### 3C. Reviews routes — `backend/src/routes/reviews.js`
Mount: `apiRouter.use('/reviews', authenticateToken, reviewsRoutes)`

| Method | Path | Description |
|--------|------|-------------|
| GET | `/user/:userId` | Get reviews for a user (paginated) |
| GET | `/user/:userId/summary` | Get average rating + count + distribution |
| POST | `/` | Create review (validates transaction exists and is COMPLETED, reviewer is participant) |
| PUT | `/:id` | Update own review |
| DELETE | `/:id` | Delete own review |

### 3D. Notifications routes — `backend/src/routes/notifications.js`
Mount: `apiRouter.use('/notifications', authenticateToken, notificationsRoutes)`

| Method | Path | Description |
|--------|------|-------------|
| GET | `/` | List user's notifications (paginated, newest first) |
| GET | `/unread-count` | Get unread count |
| PATCH | `/:id/read` | Mark as read |
| PATCH | `/read-all` | Mark all as read |
| DELETE | `/:id` | Delete notification |

### 3E. Helper: Notification creation utility
**File:** `backend/src/utils/notifications.js`
- `createNotification(userId, type, title, message, relatedId)` — inserts into notifications table
- Called from: messages route (new message), transactions route (status change), reviews route (new review)

## PHASE 5: Real-Time Chat

Since Vercel serverless doesn't support WebSockets, use **Supabase Realtime** for message subscriptions. The Supabase infrastructure already exists.

### 5A. Add Supabase Kotlin SDK dependency
**File:** `app/build.gradle.kts` + `libs.versions.toml`
- Add `io.github.jan-tennert.supabase:realtime-kt` and `io.github.jan-tennert.supabase:postgrest-kt`
- Add Ktor client engine for Android

### 5B. Create `SupabaseModule.kt` in Hilt DI
**File:** `app/src/main/java/.../di/SupabaseModule.kt`
- Provide Supabase client configured with project URL and anon key
- Enable Realtime plugin

### 5C. Create `ChatRealtimeService.kt`
**File:** `app/src/main/java/.../data/sync/ChatRealtimeService.kt`
- Subscribe to `postgres_changes` on `messages` table filtered by conversation_id
- Emit new messages as Flow to the ChatViewModel
- Handle connection lifecycle (connect on chat screen open, disconnect on leave)

### 5D. Update `ChatViewModel` to use real-time
**File:** `app/src/main/java/.../ui/chat/ChatViewModel.kt`
- On screen open: load history from API, then subscribe to real-time
- On new message received: insert into Room + update UI
- On send: POST to backend API (which inserts into DB, triggering Supabase Realtime)
